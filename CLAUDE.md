# Room Rental — Design System Repository

## Purpose

This repository is the **single source of truth** for all UI/UX standards in the Room Rental Management System.
It is consumed by every frontend application in the project.

## Repository Structure

```
tokens/tokens.json              ← W3C Design Tokens — the ONLY file humans edit for token changes
generated/                      ← ALL auto-generated by scripts/generate-tokens.js — never edit manually
  web/tokens.css                ← CSS custom properties (verbose: --color-primary-DEFAULT) — for production web apps
  prototype/base.css            ← CSS for prototype development (simplified: --color-primary) — used by all prototypes
  blazor/RoomRentalTheme.cs     ← MudBlazor 9.x theme
  mobile/tokens.json            ← flat camelCase key-value for React Native / Flutter
components/
  specs/                        ← markdown component specs (output from Claude Design)
  prototypes/                   ← Claude Design HTML exports
    shared/                     ← (Phase 2) shared atoms used by all prototypes
    auth-server/                ← auth, identity, OAuth screens
    manager-portal/             ← landlord/property manager workspace
guidelines/                     ← human-readable design rules
docs/
  app-ownership.md              ← ownership manifest: which screen/component belongs to which app
scripts/
  generate-tokens.js            ← Node.js generator — no external deps
  audit-design.js               ← design system health check
.claude/
  agents/design-sync.md        ← Claude Code subagent for design governance
  commands/do.md               ← /do slash command for design tasks
```

## Two CSS outputs — which to use?

| File | Variable names | Use in |
|------|---------------|--------|
| `generated/web/tokens.css` | `--color-primary-DEFAULT` | Production web apps (React, Angular, Vue) |
| `generated/prototype/base.css` | `--color-primary` | Prototype HTML files only |

Prototype `index.html` files import `../../../generated/prototype/base.css`.
**Never import `colors_and_type.css` — that file no longer exists.**

## Claude Design

Component specs in `components/specs/` and prototypes in `components/prototypes/` are produced by Claude Design (not by any local agent).

When specifying components or screens:
- Always reference token names from `tokens/tokens.json`, never raw hex or px values
- Use `color.primary.DEFAULT`, `spacing.4`, `border.radius.md` notation in specs
- In prototype CSS: use simplified vars (`var(--color-primary)`, `var(--sp-4)`, `var(--radius-md)`)
- In production CSS: use verbose vars (`var(--color-primary-DEFAULT)`, `var(--spacing-4)`)
- In MudBlazor Razor: use `Color.Primary`, `@_theme.PaletteLight.Primary`

## Token Naming Convention

Tokens follow dot-notation in `tokens.json`, converted differently per target:

| tokens.json path | Production CSS | Prototype CSS | C# (MudBlazor) | Mobile JSON |
|---|---|---|---|---|
| `color.primary.DEFAULT` | `--color-primary-DEFAULT` | `--color-primary` | `Primary = "..."` | `colorPrimaryDEFAULT` |
| `color.secondary.DEFAULT` | `--color-secondary-DEFAULT` | `--color-secondary` | `Secondary = "..."` | `colorSecondaryDEFAULT` |
| `spacing.4` | `--spacing-4` | `--sp-4` | padding: `16px` | `spacing4` |
| `border.radius.md` | `--border-radius-md` | `--radius-md` | `DefaultBorderRadius` | `borderRadiusMd` |
| `color.text.primary` | `--color-text-primary` | `--fg-1` | `TextPrimary = "..."` | `colorTextPrimary` |

The mapping from token keys to prototype CSS vars is defined in `scripts/generate-tokens.js` → `PROTOTYPE_ALIAS`.

## Workflow

1. Edit `tokens/tokens.json` for any design token change
2. Run `node scripts/generate-tokens.js` — regenerates ALL `generated/` files
3. Run `node scripts/audit-design.js` — verify no issues
4. Commit `tokens.json` + all `generated/` files together in one commit
5. App teams update their reference to pick up the new tokens

## Rules for This Repo

- **Never** edit files under `generated/` manually — always regenerate from `tokens.json`
- **Never** hardcode hex colors or pixel values in component specs — use token names
- **Never** create a manually-maintained CSS/JS file that duplicates token values — add to tokens.json instead
- Every breaking change (token renamed, removed, or role changed) must bump the **major** version in `$metadata.version`
- Add a CHANGELOG entry for every merge to main
- Integration guide for each stack (Blazor, Vue 3, React) is in `docs/integration-guide.md` (VI) and `docs/integration-guide.en.md` (EN)
