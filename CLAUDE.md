# Room Rental ‚Äî Design System Repository

## Purpose

This repository is the **single source of truth** for all UI/UX standards in the Room Rental Management System.
It is consumed by every frontend application in the project.

## Repository Structure

```
tokens/tokens.json              ‚Üê W3C Design Tokens ‚Äî the ONLY file humans edit for token changes
generated/                      ‚Üê ALL auto-generated by scripts/generate-tokens.js ‚Äî never edit manually
  web/tokens.css                ‚Üê CSS custom properties (verbose: --color-primary-DEFAULT) ‚Äî for production web apps
  prototype/base.css            ‚Üê CSS for prototype development (simplified: --color-primary) ‚Äî used by all prototypes
  blazor/RoomRentalTheme.cs     ‚Üê MudBlazor 9.x theme
  mobile/tokens.json            ‚Üê flat camelCase key-value for React Native / Flutter
components/
  specs/                        ‚Üê markdown component specs (output from Claude Design)
  prototypes/                   ‚Üê Claude Design HTML exports
    shared/                     ‚Üê (Phase 2) shared atoms used by all prototypes
    auth-server/                ‚Üê auth, identity, OAuth screens
    manager-portal/             ‚Üê landlord/property manager workspace
guidelines/                     ‚Üê human-readable design rules
docs/
  app-ownership.md              ‚Üê ownership manifest: which screen/component belongs to which app
scripts/
  generate-tokens.js            ‚Üê Node.js generator ‚Äî no external deps
  audit-design.js               ‚Üê design system health check
.claude/
  agents/design-sync.md        ‚Üê Claude Code subagent for design governance
  commands/do.md               ‚Üê /do slash command for design tasks
```

## Two CSS outputs ‚Äî which to use?

| File | Variable names | Use in |
|------|---------------|--------|
| `generated/web/tokens.css` | `--color-primary-DEFAULT` | Production web apps (React, Angular, Vue) |
| `generated/prototype/base.css` | `--color-primary` | Prototype HTML files only |

Prototype `index.html` files import `../../../generated/prototype/base.css`.
**Never import `colors_and_type.css` ‚Äî that file no longer exists.**

## Claude Design

Component specs in `components/specs/` and prototypes in `components/prototypes/` are produced by Claude Design (not by any local agent).

When specifying components or screens:
- Always reference token names from `tokens/tokens.json`, never raw hex or px values
- Use `color.primary.DEFAULT`, `spacing.4`, `border.radius.md` notation in specs
- In prototype CSS: use simplified vars (`var(--color-primary)`, `var(--sp-4)`, `var(--radius-md)`)
- In production CSS: use verbose vars (`var(--color-primary-DEFAULT)`, `var(--spacing-4)`)
- In MudBlazor Razor: use `Color.Primary`, `@_theme.PaletteLight.Primary`

## Token Naming Convention

Tokens follow dot-notation in `tokens.json`, converted differently per target:

| tokens.json path | Production CSS | Prototype CSS | C# (MudBlazor) | Mobile JSON |
|---|---|---|---|---|
| `color.primary.DEFAULT` | `--color-primary-DEFAULT` | `--color-primary` | `Primary = "..."` | `colorPrimaryDEFAULT` |
| `color.secondary.DEFAULT` | `--color-secondary-DEFAULT` | `--color-secondary` | `Secondary = "..."` | `colorSecondaryDEFAULT` |
| `spacing.4` | `--spacing-4` | `--sp-4` | padding: `16px` | `spacing4` |
| `border.radius.md` | `--border-radius-md` | `--radius-md` | `DefaultBorderRadius` | `borderRadiusMd` |
| `color.text.primary` | `--color-text-primary` | `--fg-1` | `TextPrimary = "..."` | `colorTextPrimary` |

The mapping from token keys to prototype CSS vars is defined in `scripts/generate-tokens.js` ‚Üí `PROTOTYPE_ALIAS`.

## Workflow

1. Edit `tokens/tokens.json` for any design token change
2. Run `node scripts/generate-tokens.js` ‚Äî regenerates ALL `generated/` files
3. Run `node scripts/audit-design.js` ‚Äî verify no issues
4. Commit `tokens.json` + all `generated/` files together in one commit
5. App teams update their reference to pick up the new tokens

## Rules for This Repo

- **Never** edit files under `generated/` manually ‚Äî always regenerate from `tokens.json`
- **Never** hardcode hex colors or pixel values in component specs ‚Äî use token names
- **Never** create a manually-maintained CSS/JS file that duplicates token values ‚Äî add to tokens.json instead
- Every breaking change (token renamed, removed, or role changed) must bump the **major** version in `$metadata.version`
- Add a CHANGELOG entry for every merge to main
- Ownership of every screen and component is tracked in `docs/app-ownership.md`
